1. Introducción

Este manual describe el uso del backend del sistema CASA, una API REST desarrollada con Node.js y Express que da soporte a la administración de contenidos multimedia, productos, ventas, paquetes y licencias, así como a la autenticación de administradores, del launcher de escritorio y de la web.

El sistema está pensado para ser consumido por:

  • Un panel de administración (frontend admin)
  • La aplicación CASA Launcher (escritorio)
  • La web CASA (licencias, multimedia, ebooks)

2. Descripción general del sistema

2.1 Propósito

El backend CASA centraliza la lógica de negocio, el acceso a la base de datos MySQL y la exposición de servicios REST y en tiempo real (Socket.IO) para gestionar ventas, paquetes, licencias, productos descargables y contenido multimedia (audios, documentos, vídeos, etc.).

2.2 Tecnologías principales

  • Node.js y Express 5 para la API REST.
  • MySQL como base de datos (pool de conexiones).
  • Socket.IO para eventos en tiempo real (subidas, eliminaciones, salas por licencia).
  • JWT para autenticación de administradores y escritorio.
  • Helmet, CORS y express-rate-limit para seguridad.
  • Multer para la subida de archivos; ExcelJS y Nodemailer según funcionalidad.

2.3 Prefijo base de las rutas

Todas las rutas de la API están bajo el prefijo /casa/. No se utiliza el segmento /v1/ en el código actual.

3. Módulo de administración (Admin)

El módulo admin (/casa/admin/) permite gestionar ventas, paquetes, licencias, pedidos, productos y multimedia. Requiere autenticación de administrador (JWT) según la configuración del frontend.

3.1 Ventas

  • Registrar venta: POST /casa/admin/registrar-venta
  • Tipos de venta: GET /casa/admin/tipos-venta
  • Listar ventas: GET /casa/admin/ventas
  • Eliminar venta: DELETE /casa/admin/ventas/:id

3.2 Paquetes

  • Registrar paquete: POST /casa/admin/registrar-paquete
  • Listar paquetes: GET /casa/admin/paquetes
  • Paquetes completos (con datos asociados): GET /casa/admin/paquetes-completos
  • Actualizar paquete: PUT /casa/admin/paquetes/:id
  • Eliminar paquete: DELETE /casa/admin/paquetes/:id

3.3 Licencias

  • Generar licencias (lote): POST /casa/admin/licencias
  • Licencias completas: GET /casa/admin/licencias-completas
  • Licencias para distribución: GET /casa/admin/licencias-distribucion
  • Actualizar estado de licencia: PUT /casa/admin/licencias/:id/status

3.4 Pedidos

  • Pedidos completos: GET /casa/admin/pedidos-completos

3.5 Filtros y conteos

  • Filtro grados: GET /casa/admin/filter-grados
  • Filtro subtipos: GET /casa/admin/filter-subtipos
  • Conteos por tipo (multimedia/productos): GET /casa/admin/countabs
  • Actualizar estado (multimedia): GET /casa/admin/update-status

3.6 Productos

  • Listar productos: GET /casa/admin/productos
  • Archivos de un producto: GET /casa/admin/productos/archivos?pro_files=... (parámetro pro_files obligatorio)
  • Actualizar producto: PUT /casa/admin/productos/:id
  • Añadir archivos a producto: POST /casa/admin/productos/:id/archivos
  • Quitar archivo de producto: DELETE /casa/admin/productos/:id/archivos/:filename
  • Eliminar producto: DELETE /casa/admin/delete-productos/:id

3.7 Subida de archivos

  • Subida multimedia: POST /casa/admin/upload?type=.... El parámetro type es obligatorio. Valores soportados: Audios, Word, Excel, PDF, AR, Videos (y otros según implementación).
  • Subida de producto: POST /casa/admin/upload-prod

3.8 Multimedia

  • Listar multimedia: GET /casa/admin/multimedia
  • Eliminar multimedia: DELETE /casa/admin/delete-multimedia

4. Login y autenticación

4.1 Login de administrador

Base: /casa/login/

  • Login: POST /casa/login/login. Se envían usuario y contraseña; el backend utiliza el procedimiento almacenado LOGIN_USUARIO_ADMIN y devuelve un token JWT para las rutas de administración.

7. CASA Launcher

Endpoints usados por la aplicación de escritorio (Launcher) para login, productos y actualizaciones.

7.1 Autenticación Launcher

Base: /casa/launcher/auth/

  • Login: POST /casa/launcher/auth/login (usuario; en esta ruta no se usa JWT en la descripción actual).

7.2 Productos Launcher

Base: /casa/launcher/productos/

9. Configuración y variables de entorno

La aplicación se configura mediante un archivo .env en la raíz del proyecto (no incluido en el repositorio por seguridad).

9.1 Variables principales

VariableUso
PORTPuerto del servidor HTTP
NODE_ENVdevelopment o production (configuración de BD y mensajes de error)
ALLOWED_ORIGINSOrígenes CORS y Socket.IO (separados por comas)
CASA_HOST_DEV, CASA_PORT_DEV, CASA_DATABASE_DEV, CASA_USERNAME_DEV, CASA_PASSWORD_DEVBase de datos en desarrollo
CASA_HOST, CASA_PORT, CASA_DATABASE, CASA_USERNAME, CASA_PASSWORDBase de datos en producción
JWT_SECRETClave para firmar el JWT del login admin
CASA_LAUNCHER_JWT_SECRET / CALA_JWT_SECRETClave JWT para el Launcher
UPLOAD_BASE_PATHRuta base de subida de archivos y estáticos en producción (por defecto /var/www/html)

9.2 Archivos estáticos

Los archivos estáticos se sirven en la ruta /FILES/static. En producción el directorio físico es el indicado por UPLOAD_BASE_PATH; en desarrollo se usa una ruta relativa al proyecto.

10. Seguridad y buenas prácticas

  • Helmet: Cabeceras de seguridad HTTP (CSP, COOP, Permissions Policy).
  • CORS: Solo se permiten orígenes definidos en ALLOWED_ORIGINS.
  • Rate limiting: Límite de 1000 solicitudes por 15 minutos por IP.
  • Validación: express-validator en los datos de entrada.
  • JWT: No exponer JWT_SECRET ni claves similares; usar valores fuertes en producción.

12. Comunicación en tiempo real (Socket.IO)

El backend emite y escucha eventos Socket.IO para notificar subidas, eliminaciones y salas por licencia.

  • client-upload-complete → el servidor emite new-upload a la sala global.
  • client-complete → el servidor emite new-issue a la sala global.
  • data-delete → el servidor emite new-delete a la sala global.
  • join-license-room / leave-license-room: salas por sesión para licencias.

Socket.IO está configurado con recuperación de estado de conexión (hasta 2 minutos).

13. Manejo de errores

Las respuestas de error de la API siguen un formato JSON estructurado:

{
  "success": false,
  "status": 500,
  "message": "Mensaje de error",
  "stack": "Stack trace (solo en desarrollo)"
}

Se gestionan errores síncronos críticos, promesas rechazadas, advertencias, errores HTTP, de Socket.IO y de Express. Las rutas no encontradas devuelven 404 en formato JSON.
